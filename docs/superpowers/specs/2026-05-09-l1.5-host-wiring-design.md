# L1.5a — Host Linux Wiring Design

**Date:** 2026-05-09
**Status:** Draft (brainstorming output, awaiting user spec review)
**Predecessor:** L1 platform crates complete (`phase-l1-platform-crates-complete` tag, master + `phase-l1-linux-poc` branch)
**Successor:** L1.5b (viewer Linux wiring + Linux↔Linux end-to-end smoke) — separate plan, TBD
**Approach:** Approach 2 — factory-centric refactor; `VideoEncoderBackend` enum closed inside `platform/win/`; `lib.rs` main loop becomes cfg-transparent via `platform::*` re-exports.

---

## 1. Goal

Refactor `crates/host/src/lib.rs` so that the cross-platform plumbing (UDP transport, signaling, watchdog, audio routing, filetransfer, clipboard polling, keepalive) is decoupled from the Windows-specific guts (D3D11 device, DXGI Desktop Duplication, NVENC / MF encoders, `prdt_input_win` free functions). After the refactor, `prdt host` builds and runs end-to-end on WSL2 Ubuntu using the L1 platform crates (`prdt-media-linux` + `prdt-input-linux`). Linux↔Linux end-to-end smoke is **not** part of L1.5a — it requires viewer wiring (L1.5b). L1.5a verifies host-side correctness via in-process tokio test (mock viewer transport).

## 2. Non-goals

- viewer Linux wiring (`crates/viewer/src/lib.rs` 2029 lines) — deferred to **L1.5b**
- Linux↔Linux end-to-end smoke (the original 9-step checklist from L1) — runnable only after L1.5b
- Wayland portal capture / libei / wl-clipboard — L2
- HW codecs on Linux (VAAPI / NVENC / NVDEC) — L2 / L3
- `Cmd::Gui` execution on Linux — L2 (gui-host stays Windows-only at runtime; build-green target only)
- cross-OS scancode normalization (Win viewer ↔ Linux host) — L2 wire-protocol fix
- multi-monitor non-zero-origin support — L2
- audio device autodetection improvements on WSLg — L2

## 3. Background

L1 landed `prdt-media-linux` and `prdt-input-linux` with full implementations and 29 unit tests passing. The L1 plan T11 had assumed `crates/host/src/lib.rs` was already `#[cfg(windows)]` sibling-friendly. Reality: every key import (`D3d11Device`, `DxgiNvencProducer`, `HwHevcEncoder`, `MfH265Encoder`, `prdt_input_win::*`, `dxgi_sw_producer::DxgiSwProducer`, `encoder_dispatch::VideoEncoderBackend`) is unguarded. Realistic L1.5a is therefore a refactor task — the platform crates are ready, but the host needs to grow a `platform/` module to consume them.

`crates/host/src/` source files audit (master @ post-L1):

| File | Lines | Windows-coupled? |
|---|---|---|
| `lib.rs` | 1034 | Mixed: cross-platform plumbing + ~200 lines Windows-specific |
| `encoder_dispatch.rs` | ~70 | Yes (`Hw(HwHevcEncoder)` from media-win) |
| `dxgi_sw_producer.rs` | 211 | Yes (D3D11 + media-win + media-sw mix) |
| `status.rs` | — | No |
| `watchdog.rs` | — | No |

The factory-centric refactor moves the two Windows-coupled standalone files into `platform/win.rs` and adds a small `platform/linux.rs` that consumes the L1 platform crates' free-function and producer APIs.

## 4. Architecture overview

```
                 ┌────────────────────────────────────────┐
                 │  prdt unified bin (crates/client)      │
                 │  Cmd::Host on Linux + Windows          │
                 │  Cmd::Gui Windows only                 │
                 └────────────────────────────────────────┘
                               │
                               ▼
                 ┌────────────────────────────────────────┐
                 │  prdt-host (lib)                       │
                 │  lib.rs: main loop (cross-platform)    │
                 │  status.rs / watchdog.rs (cross-pf)    │
                 │  platform/                             │
                 │    mod.rs   — cfg-aliased re-exports   │
                 │    win.rs   — Windows guts (moved      │
                 │               from encoder_dispatch +  │
                 │               dxgi_sw_producer +       │
                 │               input/clipboard wrappers)│
                 │    linux.rs — thin wrappers around     │
                 │               prdt-media-linux +       │
                 │               prdt-input-linux         │
                 └────────────────────────────────────────┘
                       │                              │
                       ▼ (Win)                        ▼ (Linux)
              prdt-media-win                  prdt-media-linux
              prdt-media-sw                   prdt-media-sw
              prdt-input-win                  prdt-input-linux
                       │                              │
                       └─── prdt-protocol::VideoProducer ─┘
                       └─── cross-platform: prdt-transport / prdt-crypto /
                            prdt-audio / prdt-filetransfer /
                            prdt-signaling-* / prdt-nat-traversal
```

`lib.rs` body imports everything Windows-or-Linux it needs through `platform::*`. Internally, `platform/mod.rs` is `#[cfg(windows)] pub use win::*; #[cfg(target_os = "linux")] pub use linux::*;`. Symbol names are identical across cfg, so `lib.rs` is fully cfg-transparent.

Existing `[target.'cfg(windows)'.dependencies]` block is unchanged.

### `crates/host/src/` file structure

| File | Action |
|---|---|

| `platform/linux.rs` | **Create**. | `encoder_dispatch.rs` | **Delete** (content moved to `platform/win.rs`). |
| `dxgi_sw_producer.rs` | **Delete** (content moved to `platform/win.rs`). |
| `status.rs`, `watchdog.rs` | **No change**. |

### `platform::*` API surface (the cfg-transparent boundary)

```rust
// re-exported from platform/mod.rs
pub fn build_video_producer(
    args_encoder: &str,            // "auto" | "nvenc" | "mf" | "openh264"
    output: &OutputDescriptor,
    bitrate_bps: u32,
    fps: u32,
    negotiated_codec: prdt_protocol::Codec,
) -> anyhow::Result<(Box<dyn prdt_protocol::VideoProducer>, &'static str)>;
//                                                          └── backend_name (separate channel
//                                                              so we don't grow the trait)

pub fn pick_default_output(args: &Args) -> anyhow::Result<OutputDescriptor>;
//   Windows: enumerates DXGI adapters + outputs, returns selected OutputInfo
//   Linux:   X11 root window's Geometry, returned as a unit-equivalent placeholder

pub fn dispatch_input(event: prdt_protocol::InputEvent) -> Result<(), DispatchError>;
pub fn read_clipboard_text() -> Result<String, ClipboardError>;
pub fn write_clipboard_text(text: &str) -> Result<(), ClipboardError>;
pub fn clipboard_sequence_number() -> u32;
pub fn virtual_desktop_rect() -> prdt_protocol::MonitorRect;
//   Linux: first call also runs prdt_input_linux::uinput_injector::init_with_geometry(w, h)
//          so the uinput ABS range is set before any inject_event() runs.

pub const MAX_CLIPBOARD_BYTES: usize;
pub type OutputDescriptor;       // cfg-conditional alias, defined per OS in win.rs / linux.rs
                                 // (see §7 for the concrete definitions)
```

## 6. `lib.rs` edit plan

### Imports (line 18-30 today)

```rust
// REMOVE:
use prdt_input_win::{
    clipboard_sequence_number, read_clipboard_text, virtual_desktop_rect, write_clipboard_text,
    SendInputInjector, MAX_CLIPBOARD_BYTES,
};
use prdt_media_sw::{Openh264Encoder, Openh264EncoderConfig};
use prdt_media_win::{
    dxgi::enumerate_outputs_for_adapter, pick_default_adapter, D3d11Device, DxgiNvencProducer,
    HwHevcEncoder, MfH265Encoder, NvencEncoder, NvencEncoderConfig,
};
use dxgi_sw_producer::DxgiSwProducer;
use encoder_dispatch::VideoEncoderBackend;
```

```rust
// ADD:
mod platform;
use platform::{
    build_video_producer, clipboard_sequence_number, dispatch_input, pick_default_output,
    read_clipboard_text, virtual_desktop_rect, write_clipboard_text, MAX_CLIPBOARD_BYTES,
};
// Note: Windows-specific types like SendInputInjector / D3d11Device / NvencEncoder / etc.
// are no longer referenced from lib.rs — they live in platform/win.rs only.
```

### Output enumeration (line ~440-460 today)

Replace the explicit `pick_default_adapter` + `enumerate_outputs_for_adapter` chain with:
```rust
let output = pick_default_output(&args).context("pick_default_output")?;
```

### Producer construction (line 462-471 today)

Replace `pick_encoder` + `match VideoEncoderBackend` block with:
```rust
let (mut producer, backend_name) = build_video_producer(
    args.encoder.as_ref(),
    &output,
    bitrate_bps,
    fps,
    req.codec,
).context("build_video_producer")?;
info!(backend = backend_name, codec = req.codec.name(), "encoder ready");
```

The `pick_encoder` function and the `VideoEncoderBackend` enum move into `platform/win.rs`. Linux doesn't need them — `platform/linux.rs::build_video_producer` directly returns a `LinuxSwProducer` boxed.

### Input injection sites

Wherever `lib.rs` currently calls `SendInputInjector::send(event)` (or the equivalent free function), replace with `dispatch_input(event)?`. The Linux `dispatch_input` calls `prdt_input_linux::inject_event(event)?` and maps the error.

### Clipboard polling sites (line 674-691 today)

No source-level change — `clipboard_sequence_number()`, `read_clipboard_text()`, `write_clipboard_text(text)` are now imported from `platform::*` (cfg-aliased). The same call signatures work.

### Geometry usage

Same: `virtual_desktop_rect()` is cfg-aliased. **First call on Linux** triggers internal `prdt_input_linux::uinput_injector::init_with_geometry(rect.width(), rect.height())` so the uinput ABS range is locked before any `dispatch_input` runs. Idempotent thereafter.

### `Cmd::Gui` arm (in `crates/client/src/main.rs`)

Stays `#[cfg(windows)]`. Linux invocation of `prdt gui` returns `Err(anyhow::anyhow!("Cmd::Gui not yet implemented on Linux (L2)"))`.

### CLI flag normalization

`Args.encoder: String` accepts arbitrary input. On Linux:
```rust
fn normalize_encoder(arg: &str) -> &'static str {
    match arg {
        "openh264" | "auto" => "openh264",
        "nvenc" | "mf" => {
            tracing::warn!(requested = arg, "Linux SW codec only; falling back to openh264");
            "openh264"
        }
        other => {
            tracing::warn!(requested = other, "unknown encoder; falling back to openh264");
            "openh264"
        }
    }
}
```

This lives inside `platform/linux.rs` and is invoked at the top of `build_video_producer`.

## 7. `OutputDescriptor` type alias

Type-alias approach (case A from Block A questions):

```rust
// platform/win.rs
pub type OutputDescriptor = prdt_media_win::OutputInfo;

// platform/linux.rs
pub type OutputDescriptor = ();   // X11 root is implicit; no descriptor needed
```

`pick_default_output` returns the type-aliased value. `lib.rs` is structurally cfg-transparent — it just stores `let output = pick_default_output(&args)?;` without inspecting fields. `build_video_producer` takes `&OutputDescriptor`, ignored on Linux.

## 9. Error model

### `platform::DispatchError` (new, in `platform/mod.rs`)

```rust
#[derive(Debug, thiserror::Error)]
pub enum DispatchError {
    #[error("input dispatch backend error: {0}")]
    Backend(String),
}
```

Both Windows and Linux platforms map their backend errors into `DispatchError::Backend(e.to_string())`. Cheap, lossless for diagnostic purposes.

### `platform::ClipboardError` (new, in `platform/mod.rs`)

```rust
#[derive(Debug, thiserror::Error)]
pub enum ClipboardError {
    #[error("clipboard backend error: {0}")]
    Backend(String),
    #[error("no text content available")]
    NoText,
    #[error("clipboard payload too large: {0} bytes")]
    TooLarge(usize),
}
```

Same shape as `prdt_input_core::ClipboardError`. Both Windows (`prdt_input_win::ClipboardError`) and Linux (`LinuxInputError`) map into this. `lib.rs` only sees this common type.

### `build_video_producer` returns `anyhow::Result<...>`

Both Windows and Linux paths are wrapped in `anyhow::Error` for diagnostic flexibility. Caller chains `.context(...)` as needed.

## 13. Open questions for the implementer

- **`OutputDescriptor` type alias**: the `pub type OutputDescriptor = ();` on Linux works but feels stylistically empty. If the implementer prefers, an opaque `pub struct OutputDescriptor { _phantom: PhantomData<()> }` with `impl Default` is acceptable. Decide at code-write time; behavior identical.
- **`backend_name()` return**: tuple `(Box<dyn VideoProducer>, &'static str)` keeps the `VideoProducer` trait minimal but is slightly awkward. The cleaner alternative is to add `fn backend_name(&self) -> &'static str;` to `prdt_protocol::VideoProducer`, which is a 1-line trait extension that the existing impls (`DxgiNvencProducer`, `DxgiSwProducer`, `LinuxSwProducer`) can all satisfy trivially. **Recommendation**: extend the trait — cleaner, future-proof, single source of truth.
- **`Cmd::Gui` Linux behavior**: spec says error-exit, but `prdt host` could be a reasonable fallback (drop into headless mode). Decide based on implementer judgment; error-exit is safer for L1.5a.
- **`linux_startup_smoke.rs` mock transport**: should reuse `prdt-transport`'s `InProcTransport` if it exists at the `prdt_transport` public API; otherwise stand up a minimal mpsc-channel-backed transport just for this test. Implementer chooses based on existing helpers.
- **Linux `Ctrl-C` shutdown path**: existing `tokio::signal::ctrl_c()` should work, but verify on WSLg (signal forwarding through WSL2 init can be quirky). If broken, document a SIGTERM alternative.
