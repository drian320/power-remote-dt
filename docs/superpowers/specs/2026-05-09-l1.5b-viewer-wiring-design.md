## §1. Goal / Scope / Non-goals

### Goal
`prdt-viewer` を Linux で動作させ、**Linux↔Linux end-to-end smoke** (host: Linux WSLg / viewer: Linux WSLg) を通す。`prdt connect <host_id>` が winit ウィンドウを開き、フレーム表示 + マウス/キーボード/クリップボード伝送ができる。

### Architecture
L1.5a と同じ **Approach 2** (factory-centric)。L1.5b 内では **Approach A** (closed enum + cfg variants):
- `crates/viewer/src/{platform/mod.rs, platform/win.rs, platform/linux.rs}` を新設
- `lib.rs` 先頭の `#![cfg(windows)]` を削除し、Windows-specific items を個別 `#[cfg(windows)]` で gate
- `LatestFrame` / `ViewerConsumer` / `ViewerRenderer` enum は `PlatformFrame` / `PlatformConsumer` / `PlatformRender` に統一・移管 (lib.rs から opaque で扱う)
- Linux 描画は softbuffer + `prdt_media_linux::i420_to_bgra::i420_to_bgra` (既存) で実装

### In scope
- Linux SW H.264 decode (`prdt_media_sw::Openh264Decoder`、cross-platform) + softbuffer 描画
- マウス/キーボード event capture from winit → wire `InputEvent` → host (host 側は L1.5a で uinput 経由 inject 済み)
- クリップボード sync (`prdt_input_linux` の `read_clipboard_text` / `write_clipboard_text` / `clipboard_sequence_number`)
- `crates/client/src/main.rs` の Linux dispatcher で `Cmd::Connect` / `Cmd::Viewer` を `prdt_viewer::run_with_args` にルート (L1.5a で stub になっていた)
- `crates/viewer/tests/linux_connect_smoke.rs` `#[ignore]`-gated 統合 smoke
- 9-step manual smoke checklist (host + viewer 両方 Linux WSLg)

### Out of scope (L2 以降)
- **Wayland**: X11 のみ (XWayland 経由で WSLg は X11 として見えるので OK)
- **Linux 上 HW codec**: VAAPI / NVENC / NVDEC は無し、openh264 SW のみ
- **Overlay child process** (`prdt-viewer-overlay`): Linux では無効、ESC は即時切断 (`event_loop.exit()`) に bind
- **Audio**: デフォルト off。`--audio on` で opt-in (`libasound2-dev` 要)。Windows 側 default は変更なし
- **File transfer の smoke 検証**: コードパスは有効 (winit `DroppedFile` + `prdt_filetransfer` は cross-platform)、明示的な drag-drop テストは smoke list から除外
- **Cross-OS scancode normalization**: winit の `PhysicalKeyExtScancode` は Linux/Windows で異なる evdev/scancode 値を返す可能性 — Linux↔Linux なら一貫しているので smoke 通る
- **multi-monitor non-zero-origin**: single primary monitor のみ
- **D3D11 device-removed 相当の Linux recovery**: softbuffer は failure mode が違う (X11 connection 切断系のみ)、L1.5b では `RenderError::DeviceLost` を Linux 側は返さない

- 手動: WSLg 上で `prdt host` (端末 A) + `prdt connect <host_id>` (端末 B) でフレーム + マウス + キーボード + クリップボードが動く

## §4. `LinuxRenderer` 詳細 — softbuffer + I420→BGRA

### softbuffer の I/O contract

softbuffer 0.4 の最小 lifecycle:
```rust
let ctx = softbuffer::Context::new(window.clone())?;
let mut surface = softbuffer::Surface::new(&ctx, window.clone())?;
surface.resize(NonZeroU32::new(w)?, NonZeroU32::new(h)?)?;
let mut buffer = surface.buffer_mut()?;     // SurfaceBuffer<[u32]>
buffer[i] = 0x00RRGGBB;                     // BE notation; LE 上は B,G,R,X 順
buffer.present()?;
```

`prdt_media_linux::i420_to_bgra` は既に `out_bgra[off]=B, off+1=G, off+2=R, off+3=0xFF` の順で書く (`i420_to_bgra.rs` の docstring が明示)。LE 環境で `&mut [u8]` を `&mut [u32]` として読むと `0xFFRRGGBB` (alpha=0xFF) になり softbuffer 期待と完全一致。**Linux x86_64 / ARM64 はどちらも LE なので問題なし**。L2 で BE platform を扱うなら別 path 必要。

### `PlatformRender` lifecycle

```rust
pub fn build_render(window: Arc<Window>, w: u32, h: u32) -> Result<PlatformRender, RenderError> {
    let ctx = softbuffer::Context::new(Arc::clone(&window))
        .map_err(|e| RenderError::Init(format!("softbuffer::Context::new: {e}")))?;
    let mut surface = softbuffer::Surface::new(&ctx, Arc::clone(&window))
        .map_err(|e| RenderError::Init(format!("softbuffer::Surface::new: {e}")))?;
    let nz_w = NonZeroU32::new(w.max(1)).unwrap();
    let nz_h = NonZeroU32::new(h.max(1)).unwrap();
    surface.resize(nz_w, nz_h)
        .map_err(|e| RenderError::Init(format!("Surface::resize: {e}")))?;
    Ok(PlatformRender {
        window, _ctx: ctx, surface,
        scratch_bgra: vec![0u8; (w * h * 4) as usize],
        last_size: (w, h),
    })
}
```

### `present_frame` の hot path (60 Hz)

```rust
pub fn present_frame(r: &mut PlatformRender, f: &PlatformFrame, _decoder_label: &str)
    -> Result<(), RenderError>
{
    let PlatformFrame::I420(i420) = f;
    let stream_w = i420.width;
    let stream_h = i420.height;

    // surface_size = stream_size を強制 (host 側にウィンドウ resize で
    // 解像度変更要求は出さない、L1.5b は letterbox/scale 無し)。
    if r.last_size != (stream_w, stream_h) {
        let nz_w = NonZeroU32::new(stream_w.max(1)).unwrap();
        let nz_h = NonZeroU32::new(stream_h.max(1)).unwrap();
        r.surface.resize(nz_w, nz_h)
            .map_err(|e| RenderError::Present(format!("Surface::resize: {e}")))?;
        r.scratch_bgra.resize((stream_w * stream_h * 4) as usize, 0);
        r.last_size = (stream_w, stream_h);
    }

    // I420 -> BGRA conversion (CPU loop, ~6ms @ 1080p on a modern x86_64).
    i420_to_bgra(i420, &mut r.scratch_bgra);

    // softbuffer の u32 buffer に流す。
    let mut buf = r.surface.buffer_mut()
        .map_err(|e| RenderError::Present(format!("buffer_mut: {e}")))?;
    debug_assert_eq!(buf.len() * 4, r.scratch_bgra.len());
    let buf_bytes: &mut [u8] = bytemuck::cast_slice_mut(&mut buf);
    buf_bytes.copy_from_slice(&r.scratch_bgra);

    buf.present()
        .map_err(|e| RenderError::Present(format!("Surface::present: {e}")))?;
    Ok(())
}
```

### Frame lifecycle (worker task → render thread)

L1.5a host の producer 形と対称:
1. **worker task** (tokio): wire packet 受信 → `PlatformConsumer::Openh264.decoder.decode(packet)` → `Option<I420Frame>`
2. 出力 `I420Frame` を `Arc::new` で wrap → `ViewerShared.latest_frame: Mutex<Option<(PlatformFrame, u64)>>` に書く (古いフレームは捨てる latest-wins、既存 Windows と同じ)
3. **render thread** (winit `about_to_wait` → `request_redraw` → `RedrawRequested` → `render_frame()`): `latest_frame` を pop → `platform::present_frame(...)` を呼ぶ

decoder は `prdt_media_sw::Openh264Decoder` (cross-platform) を直接使用、L1 の `LinuxSwDecoder` wrapper は viewer では使わない (host 用 trait 適合のための thin wrapper)。

### エラー処理

softbuffer は driver-level の "device lost" 概念を持たない:
- X11 connection 切断 → `Surface::buffer_mut()` が `Err(SoftBufferError::PlatformError(_))` 返す
- ウィンドウ destroy 後に呼ぶと panic ではなく Err
- compositor crash / display detach 系も同様にエラー伝搬

→ L1.5b では Linux 側 `present_frame` は `RenderError::DeviceLost` を返さず、全部 `Present` で wrap。lib.rs の `should_exit` ロジックは Windows-only にとどまる (上記 §3 lib.rs 例参照)。

## §5. Cross-cutting changes + error model + testing

### 5.1 Codec gating (Linux viewer は H.264 のみ)

- `Args.codec` 既定は `auto` だが、Linux viewer の `viewer_supported_codecs` は wire 上 `vec![Codec::H264]` を送る (Hello に積む)
- `choose_decoder` の Linux 実装:
  - `("openh264" | "auto", Codec::H264)` → `DecoderChoice::Openh264`
  - `("nvdec" | "mf", _)` → 早期 Err: `"--decoder {nvdec|mf} requested on Linux but no HW decode available; use --decoder openh264 or --decoder auto"`
  - `(_, Codec::H265)` → Err: `"host negotiated H.265 but Linux viewer has no H.265 decoder; ask host to advertise H.264"`
  - `(_, Codec::Av1)` → Err (既存と同じ)
- 実装方針: `choose_decoder` は cross-platform に保ち (引数 + 戻り値が portable)、内部の `match (decoder_arg, negotiated)` の各 arm を `#[cfg]` ガード — Linux では HW arm が unreachable、Windows では既存ロジック

### 5.2 Input mapping の portability 確認

- `physical_key_to_scancode(key: PhysicalKey) -> Option<u32>` は `winit::platform::scancode::PhysicalKeyExtScancode` 経由 — Linux で取れる scancode は Linux evdev 値、Windows は Windows scancode、**完全には一致しない** (cross-OS 接続時に host 側でズレる)
- 既知制約: cross-OS scancode normalization は L2 (spec out-of-scope)。Linux↔Linux なら一貫しているので smoke 通る
- `RawInputCapturer::map_winit_mouse_button` は完全 portable — `platform/input_map.rs` に移動

### 5.3 Disconnect path (overlay 無し版)

- `WindowEvent::KeyboardInput` の ESC 処理:
  - Windows: `overlay.spawn_if_idle()` (既存)
  - Linux: `event_loop.exit()` を直接呼ぶ ("Esc = quick disconnect")
- cfg gate で 2 分岐
- 通常切断: ウィンドウの ✕ ボタン (`WindowEvent::CloseRequested`) と `Ctrl+C` (signal handler) は OS 共通で既存通り

### 5.4 ViewerApp struct の cfg-gating

- `dev: D3d11Device` フィールド → 削除し `PlatformRender` 内に内包
- `decoder: String` → そのまま (label 用 string)
- `overlay: Option<OverlaySupervisor>` → `#[cfg(windows)]` で gate
- `last_overlay_tick` → `#[cfg(windows)]` で gate
- `disconnect_requested` → 残す (Linux でも将来使い得る)
- `spawn_worker_tasks` の `dev: D3d11Device` 引数は削除 — Windows 内部の D3D11Device 取得は `platform::win::build_consumer` の中で `pick_default_adapter()` + `D3d11Device::create()` を呼ぶ。結果として lib.rs から D3D11 が完全に消える

### 5.5 Error model 全体像

| 種類 | Windows | Linux |
|------|---------|-------|
| Renderer init 失敗 | `RenderError::Init` (D3D11 device create 失敗 etc.) | `RenderError::Init` (softbuffer Context/Surface init 失敗) |
| Renderer present 失敗 | `RenderError::Present` (transient, log-and-continue) | `RenderError::Present` (transient, log-and-continue) |
| Device lost (致命的) | `RenderError::DeviceLost` (DXGI device-removed) | 返さない (X11 死亡は別経路で表面化) |
| Decoder init | `ConsumerError::Init` | `ConsumerError::Init` (openh264) |
| Decode 失敗 | `ConsumerError::Decode` | `ConsumerError::Decode` |
| Clipboard | `ClipboardError::*` (3 variant、L1.5a host と同一) | 同上 |

lib.rs はエラーを直接 OS-specific 型として扱わない; `platform::*Error` のみで扱う。

## §7. Open questions / 後続フォロー

L1.5b スコープ外、L1.5c / L2 で扱うべき項目:
- **viewer cooperative shutdown**: 現在 host/viewer 両方 spawn_blocking + abort 経路で詰める可能性あり。`CancellationToken` を `run_with_args` 全体に通して `prdt::cancel()` で graceful shutdown を実装 (L1.5c)
- **Cross-OS scancode normalization**: wire-protocol レベルで evdev/winit/Windows scancode の対応表を導入 (L2)
- **multi-monitor non-zero-origin**: virtual desktop の左上座標が負の時の uinput ABS offset 対応 (L2)
- **HW decode on Linux**: VAAPI / NVDEC バックエンドの platform/linux.rs 拡張 (L2)
- **Wayland**: softbuffer は Wayland も話せるが、portal screen capture は host 側、libei は input、wl-clipboard は clipboard で別 crate になる (L2)
- **`Cmd::Gui` Linux 対応**: GUI front-end (egui or similar) を Linux 上で動かす (L2)
- **Audio default-on on Linux**: WSLg の PulseAudio 動作確認後 (L2)
- **`prdt_input_win::RawInputCapturer::map_winit_mouse_button` の cleanup**: L1.5b で `platform::input_map` に移動した後、win 側の同名関数は呼び出し側がいなくなる。削除は L2 cleanup へ deferral

## §8. Pre-mortem — 想定失敗 mode

1. **softbuffer の Surface generic 制約**: softbuffer 0.4 の `Surface<D, W>` は `D: HasDisplayHandle, W: HasWindowHandle`。winit 0.30 の `Arc<Window>` は両方実装するが、unsafe trait なので `D = W = Arc<Window>` で実体化できることを T5 で型 check。失敗時は `D = Arc<Window>, W = &'static Window` に変更する逃げ道あるが lifetime 問題があり面倒
2. **bytemuck::cast_slice_mut の alignment**: softbuffer の `&mut [u32]` は 4 バイト alignment 保証あり、`&mut [u8]` への cast_slice_mut も safe (bytemuck のドキュメント保証)。仮に panic した場合は unsafe pointer cast に逃げる
3. **i420_to_bgra の output が softbuffer 期待と microorder ずれている**: alpha が 0xFF 固定 → softbuffer は alpha を無視するので問題なし。BT.709 vs BT.601 の色域違いは smoke 段階では visible だが許容 (L2 で改善)
4. **WSLg X11 で softbuffer が SHM を要求して失敗**: softbuffer は SHM 必須ではない (XPutImage fallback あり) が、性能は落ちる。WSLg は MIT-SHM サポート済みなので OK のはず
5. **clipboard polling 競合**: viewer と host が同じ X11 _CLIPBOARD selection を交互に上書きする loop 防止は既存の sequence_number + `last_set_text` echo-suppress ロジックで OK (L1.5a 実装済み)。Linux 版は `prdt_input_linux::clipboard_sequence_number` が thread-local owner thread に bump される設計
6. **lib.rs:1843 の test mod が D3D11 を直接触る**: `#[cfg(all(test, windows))]` で全体を gate するか、D3D11 を触らないテストだけ残してそれ以外を cfg gate するか — T1 で精査
7. **`prdt-viewer` crate を Linux client に追加すると build 時間悪化**: prdt-media-linux + prdt-input-linux + softbuffer 追加で +30s 程度の予測。ICEd build の妨げにはならない
8. **Linux で `prdt connect` の signaling 経路 (TURN/STUN)**: `prdt-nat-traversal` は cross-platform (tokio + std::net のみ)、`prdt-signaling-client` も cross-platform。問題なし
